# Release Notes

## Workflow

1. Identify audience: users, developers, admins, internal team, or customers.
2. Group changes into features, fixes, improvements, docs, deprecations, and breaking changes.
3. Put user impact before implementation detail.
4. Include upgrade steps and migration notes when behavior changes.
5. Mark known issues and rollback notes when relevant.

## Style

- Use concise bullets.
- Avoid internal ticket noise unless the audience needs it.
- Call out breaking changes clearly.

## Verification

- Check version, date, links, issue references, and compatibility notes.
- Confirm every breaking change has guidance.
