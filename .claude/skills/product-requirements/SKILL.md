# Product Requirements

## Workflow

1. Identify user, problem, goal, business context, constraints, and decision owner.
2. Separate goals, non-goals, assumptions, requirements, and open questions.
3. Define acceptance criteria that can be tested.
4. Include edge cases, permissions, analytics, rollout, dependencies, and support impact.
5. Keep scope explicit and mark tradeoffs.

## PRD Structure

- Summary: what is changing and why.
- Problem: user pain, evidence, affected segments.
- Goals and non-goals: boundaries for execution.
- Requirements: functional, non-functional, data, analytics, permissions.
- UX notes: key flows, states, and copy requirements.
- Release: rollout, success metrics, risks, support, rollback.

## Verification

- Check each requirement has a testable acceptance criterion.
- Confirm unresolved questions and decisions are visible.
- Align with engineering, design, data, and support needs.
